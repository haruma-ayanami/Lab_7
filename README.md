Лабораторно-практическая работа №7

Поведенческие паттерны проектирования: Наблюдатель, Стратегия и Шаблонный метод

Выполнил: Ткачев А. Д.
Группа: 2407СА1
Проверил: Макаров М. С.

Цель работы

Изучить поведенческие паттерны проектирования «Банды четырёх (GoF)» на примере разработки приложения, моделирующего систему мониторинга и оповещения о событиях. Освоить на практике паттерны Наблюдатель (Observer) для организации механизма подписки на события, Стратегия (Strategy) для динамического выбора формата сообщений и Шаблонный метод (Template Method) для определения общей структуры обработки уведомлений.
В процессе выполнения работы необходимо научиться организовывать взаимодействие между объектами системы, обеспечивать гибкость архитектуры приложения и реализовывать возможность расширения функциональности без изменения существующего кода.

Задание

Разработать консольное приложение, моделирующее работу системы мониторинга и оповещения о событиях.
Программа должна обеспечивать:
Отслеживание различных метрик системы, таких как загрузка процессора, использование оперативной памяти и сетевой активности. При превышении заданных пороговых значений должны генерироваться события.
Реализацию механизма уведомления обработчиков о наступлении событий с использованием паттерна Observer. Монитор должен уведомлять всех подписчиков автоматически при обнаружении критического состояния метрики.
Поддержку различных каналов доставки уведомлений. В качестве обработчиков необходимо реализовать вывод сообщений в консоль, запись сообщений в файл и имитацию отправки сообщений по электронной почте.
Поддержку различных форматов сообщений с использованием паттерна Strategy. Формат уведомлений должен переключаться между текстовым, JSON и HTML представлением.
Использование паттерна Template Method для организации общего алгоритма обработки событий. Базовый класс обработчика должен задавать последовательность действий при обработке уведомления, а конкретные классы обработчиков должны реализовывать отдельные шаги алгоритма.
Продемонстрировать работу программы в основной функции приложения.

Теоретическое обоснование

Поведенческие паттерны проектирования предназначены для организации взаимодействия между объектами системы и распределения ответственности между ними. Они позволяют уменьшить связанность компонентов программы и повысить гибкость архитектуры приложения.
Паттерн Наблюдатель (Observer) используется для организации зависимости типа «один-ко-многим» между объектами. В этом случае один объект, называемый издателем, уведомляет множество подписчиков об изменении своего состояния. Подписчики автоматически получают уведомления без необходимости прямого обращения к ним со стороны издателя. В контексте данной лабораторной работы монитор метрик выступает в роли издателя, а обработчики уведомлений — в роли подписчиков.
Паттерн Стратегия (Strategy) предназначен для инкапсуляции семейства алгоритмов и обеспечения возможности их взаимозаменяемости. Он позволяет изменять поведение программы во время выполнения без изменения клиентского кода. В данной работе паттерн используется для выбора различных форматов представления сообщений, таких как текстовый формат, JSON и HTML.
Паттерн Шаблонный метод (Template Method) определяет общий каркас алгоритма в базовом классе и позволяет подклассам переопределять отдельные шаги алгоритма без изменения его структуры. В рамках данной лабораторной работы данный паттерн применяется для организации последовательности обработки уведомлений обработчиками различных типов.
Использование данных паттернов позволяет обеспечить гибкость архитектуры системы мониторинга, упростить расширение функциональности приложения и уменьшить связанность между его компонентами.

Описание выполненных действий

Для выполнения лабораторной работы было разработано консольное приложение, моделирующее работу системы мониторинга и оповещения о событиях.
На первом этапе был реализован паттерн Observer. Для этого был создан класс EventMonitor, выполняющий роль издателя событий. Данный класс отслеживает значения метрик системы и при превышении пороговых значений генерирует событие OnMetricExceeded. В качестве аргументов события используется класс MetricEventArgs, содержащий информацию о типе события и данных метрики. В роли подписчиков выступают обработчики уведомлений, которые автоматически получают сообщения о возникших событиях.
На втором этапе был реализован паттерн Strategy. Для этого был создан интерфейс IFormatStrategy, определяющий метод форматирования сообщений. Далее были реализованы конкретные стратегии форматирования: TextFormatStrategy, JsonFormatStrategy и HtmlFormatStrategy. Использование данного подхода позволило обеспечить возможность динамического выбора формата сообщений без изменения кода обработчиков уведомлений.
На третьем этапе был реализован паттерн Template Method. Для этого был создан абстрактный базовый класс EventHandlerBase, определяющий общий алгоритм обработки события. В данном классе реализован шаблонный метод ProcessEvent, который задаёт последовательность действий: формирование сообщения, отправка уведомления и логирование результата обработки. Конкретные классы обработчиков ConsoleHandler, FileHandler и EmailHandler переопределяют отдельные шаги алгоритма в соответствии со способом доставки уведомлений.
В основной программе была продемонстрирована работа всех реализованных паттернов. Были созданы обработчики различных типов с различными стратегиями форматирования сообщений. Далее была выполнена проверка нескольких метрик системы, при превышении пороговых значений которых происходила генерация событий и отправка уведомлений подписчикам через разные каналы связи.

Результат выполненной работы

В результате выполнения лабораторной работы было разработано консольное приложение, моделирующее систему мониторинга и оповещения о событиях.
Проверка работы программы показала, что использование паттерна Observer позволяет автоматически уведомлять всех подписчиков о возникновении событий без жёсткой зависимости между монитором и обработчиками уведомлений.
Паттерн Strategy обеспечивает гибкость форматирования сообщений и позволяет динамически изменять способ представления уведомлений без изменения логики работы обработчиков.
Паттерн Template Method позволяет задать общий алгоритм обработки событий и обеспечить единообразную структуру обработки уведомлений различными обработчиками.
Использование данных паттернов позволило повысить гибкость архитектуры приложения, уменьшить связанность компонентов системы и обеспечить возможность дальнейшего расширения функциональности без изменения существующего кода.
